# Ball WP(Working Point) 안정화 설계

## 1. 문제 정의

### 1.1 현상
- 볼에서 거리가 멀어질수록 표시 거리가 증가함 (예: 7m인데 10m로 표시)
- 카메라를 급격하게 움직이면 LIVE 숫자가 이상하게 변함

### 1.2 거리 과대 체감의 구조적 원인
일부 ray가 카메라 앞쪽(근거리 노이즈)을 잡으면 Ball WP가 실제 볼보다 앞에 위치하게 되고, 이후 Cup과의 거리 계산에서 실제보다 더 긴 거리로 계산될 수 있음.

---

## 2. 핵심 원칙

### 2.1 Ball WP는 원점이다
- slope/aiming까지 연결된 구조에서는 Ball WP가 모든 거리 계산의 기준점(원점)
- Ball WP가 흔들리면 이후 모든 측정값이 영향을 받음

### 2.2 3×3 그리드 자체가 문제는 아님
- 3×3을 빼자는 감정적 회귀를 막아야 함
- 3×3은 "Ball FIX 안 됨" 민원 대응으로 도입된 유효한 설계

### 2.3 median vs farthest는 본질 아님
- 선택 함수의 종류가 아니라 **선택 이전의 게이트 부족**이 본질
- Ball WP를 프레임 단위로 바로 쓰는 구조가 근본 원인

---

## 3. 안정화 4요소 (단계적 파이프라인)

단일 장치(예: clamp만 적용)는 증상을 완화할 뿐이며, **군집 검증 → 대표 선택 → 표시 안정화 → 확정 누적**이 단계적으로 모두 적용되어야 근본적으로 WP 점프를 차단할 수 있음.

| 단계 | 내용 |
|------|------|
| 1. 군집 검증 | 9개 중 최소 과반(4개 이상)이 동일 군집일 때만 신뢰 가능한 표면으로 간주 |
| 2. 대표 선택 | 중앙 우선 + farthest (멀어질수록 올바른 타깃 선택) |
| 3. 표시 안정화 | clamp + smoothing (프레임당 변화량 제한) |
| 4. 확정 누적 | N프레임 누적 후 확정 (점프 제한) |

### 3.1 클러스터 최소 개수 근거
9개 중 최소 과반(4개 이상)이 동일 군집일 때만 신뢰 가능한 표면으로 간주. 3개는 소수로 노이즈에 취약하고, 5개 이상은 너무 보수적임.

---

## 4. 전략 선택지

| 옵션 | 설명 |
|------|------|
| **A. 3×3 + 안정화** | 권장. 현재 구조 유지하고 4요소 파이프라인 적용 |
| **B. 3×3 유지 + 릴리즈 안정 모드** | 보수 세팅(더 강한 clamp/smoothing) 적용 |
| **C. center-only** | 디버그/테스트 모드 전용. 릴리즈 옵션에서 제외 (이미 3×3 도입 이유가 "Ball FIX 안 됨" 민원이므로, center-only 회귀 시 성공률 저하·"아예 안 됨" 체감 리스크) |
